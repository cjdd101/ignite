点燃 · 完整设计文档 V4.0
================

**文档状态：** MVP 完整定义稿，含产品设计、交互逻辑、AI接口、数据模型、技术架构  
**最后更新：** 2026-04-27  
**设计哲学：** 存一粒火种，燃一团烈焰，蔓一片草原。  
**核心隐喻：** 灵感是火种，行动是烈焰，脉络是草原。
一、产品灵魂与边界
---------

### 1.1 解决的根本问题

现代人的大量空闲时间被碎片化信息流吞噬。这些信息彼此孤立、没有归属，刷完即忘，造成一种深层的**虚无感**——明明在接收，却什么也没有留下。

虚无感的根源是**缺乏链接**：信息与信息之间、自己与信息之间，没有形成有意义的、可回忆的结构。

### 1.2 解决路径

点燃通过三个环环相扣的动作，帮助用户把碎片编织成自己的意义网络：

1. **存火种**：灵感、念头、好奇、疑问，都可以作为一粒火种存下来。

2. **燃烈焰**：把一粒火种点燃成一团或多团烈焰——具体的、正在进行的探索行动。

3. **取新火**：烈焰燃过之后，从中保留新的火种。探索不息，火种不灭。

无数烈焰汇聚，自然蔓成**草原**——属于自己的探索脉络。草原是跨火种、跨时间的长期叙事容器。

### 1.3 产品定位与边界

点燃是一个**个人探索工具**，不是内容平台，不是社交媒体。

**MVP阶段它负责：**

* 提供动态生成的好奇火种

* 提供极度低阻力的探索启动与内容管理

* 用AI在关键节点创造火种的延续与链接

**它不负责：**

* 帮用户找到人生意义

* 生产原创内容

* 社交或分享（二期考虑）

### 1.4 核心设计原则

* **野草式生长**：用户不需要先成为建筑师。系统只提供土壤和水分（火种、结构提示），每个人自己长成草原或森林。

* **火种为货币**：火种是整个系统中唯一流通的东西。从系统来、从用户来、从烈焰的余烬中来。

* **建议而非指派**：所有AI生成的内容，都是可忽略、修改或删除的「建议」。

* **先人工，后智能**：核心机制由人设计。AI只在能创造价值且路径清晰的地方引入。

二、核心概念定义
--------

### 2.1 三种对象

| 概念     | 意象          | 本质                      | 位置                 | 来源               |
| ------ | ----------- | ----------------------- | ------------------ | ---------------- |
| **火种** | 一粒未燃的灵感种子   | 灵感、念头、问题碎片，不能直接执行       | **火盆**（灵感蓄热区）      | 用户手写、AI取火生成、系统种子 |
| **烈焰** | 一团正在燃烧的探索   | 一个正在进行的探索行动（读书、看片、游戏……） | **草原**或**野火**（行动区） | 从火种点燃而来，或用户直接创建  |
| **草原** | 一片容纳烈焰燃烧的原野 | 一个项目标题级的跨时间叙事脉络         | 草原列表               | 用户手动创建、整理时AI建议创建 |

**野火**：未归类烈焰的默认暂存区。虚拟概念，数据库查询条件为 `prairieId IS NULL`。

**草原粒度标准：** 项目标题级——不能大到一门学科，不能小到一个问题，应能容纳几团到几十团烈焰，有聚焦和蔓延空间。

### 2.2 三个动作

| 动作     | 方向         | 触发             | AI任务                                 | 用户操作                    |
| ------ | ---------- | -------------- | ------------------------------------ | ----------------------- |
| **点燃** | 火盆 → 草原/野火 | 用户对火种点击「点燃」    | ①生成多视角探索建议 ②为选定视角生成具体第一步行动 ③汇总生成烈焰草稿 | 三步向导：选视角 → 选行动 → 确认烈焰草稿 |
| **取火** | 草原/野火 → 火盆 | 烈焰完成，保存记录时自动触发 | 生成3粒灵感火种                             | 逐粒选择保留或丢弃，可重新取火         |
| **整理** | 野火 → 草原    | 用户手动点击「整理」     | 分析野火中烈焰与已有草原的匹配度                     | 分组总览 → 逐条确认归类建议或创建新草原   |

### 2.3 一个循环

text

火种（火盆） —— 点燃 ——→ 烈焰（草原/野火）
   ↑                          │
   │                          │
   └—————— 取火 ——————————————┘

**一句话：** 在火盆里存火种，点燃它成为草原上的烈焰。烈焰燃尽时取回新的火种，落回火盆。未归类的烈焰是野火，整理之后归入草原。
三、用户与场景
-------

### 3.1 用户状态模型

| 状态         | 占比   | 典型心理         | 产品应对策略                       |
| ---------- | ---- | ------------ | ---------------------------- |
| **探索欲状态**  | ~10% | 充满好奇，能主动规划   | 提供烈焰创建、火种整理、草原管理、探索页         |
| **自动驾驶状态** | ~60% | 有空闲但无明确意图    | 用火盆中的火种和草原上的烈焰构成内容流，替代碎片流    |
| **无力感状态**  | ~30% | 疲惫，抗拒一切意义和目标 | MVP暂不处理，未来通过「余烬」功能提供无压力的陪伴内容 |

### 3.2 冷启动用户旅程

**Day 1 — 邂逅第一粒火种**

用户首次打开点燃，进入**火盆**。火盆中已有系统预置的5粒火种（AI生成的知识缺口型问题），如：

* “唐僧取的真经，到底是哪几本经书？讲了什么？”

* “为什么蚊子只咬我，不咬别人？”

* “苹果手机那个‘咔哒’的解锁声，到底是谁设计的？”

用户被“蚊子”火种吸引，点击「点燃」。进入点燃三步向导：

1. **选择探索视角**：AI生成几个视角（如“看科普视频”、“读相关文章”、“自己写观察笔记”）。用户勾选了“看科普视频”和“读相关文章”。

2. **选择第一步行动**：AI为每个视角推荐具体行动——看一个5分钟科普视频、读一篇知乎高赞回答。用户确认。

3. **确认烈焰**：AI生成两团烈焰草稿，标题、推荐理由、探索口令都已填好。用户确认保存。两团烈焰进入野火。弹出平台跳转面板，用户点B站跳转看视频。

看完视频，用户回来标记完成。弹出记录弹窗（必填），用户写感想。提交后，AI取火生成3粒新火种。用户保留2粒，落入火盆。

**Day 2~4 — 收集与燃烧**

用户有时直接创建烈焰（“读《蚊子行为研究》”），有时从火盆点燃火种，有时在探索页搜索“咖啡”然后一键点燃感兴趣的火种。

**Day 5~7 — 整理成草原**

用户点击「整理」。AI发现野火中有3团烈焰都与蚊子相关，建议创建草原“蚊子与人体探秘”。用户确认，草原形成，烈焰归入。

用户也可以手动创建草原，后续将烈焰归入。

**一个月后**：用户拥有几片草原，一个流动的火盆，和一个不断生长的探索网络。

### 3.3 三种状态下的日常使用

#### 状态一：探索欲来了（~10%）

* 在火盆里写火种，或直接创建烈焰。

* 点燃火盆中的火种，通过三步向导生成烈焰。

* 在探索页输入感兴趣的关键词，AI生成相关火种，一键点燃或加入火盆。

* 手动触发「整理」，让AI帮忙归类野火中的烈焰。

* 浏览草原，回顾探索历程和时间轴。

#### 状态二：自动驾驶状态（~60%）

* 打开应用，刷火盆里的火种，或看草原上的烈焰。

* 被某条推荐理由打动，点击链接跳转继续燃烧（读书、看片）。

* 标记烈焰完成，写感想，取火，选留新火种。

#### 状态三：无力感模式（~30%）

* MVP不实现。未来「余烬」：收藏的安心老歌、老电影，纯展示，无压力。

### 4.2 核心流程

#### 流程A：点燃（火种 → 烈焰）

1. 用户在火盆或探索页对火种点击「点燃」。

2. 进入点燃向导：
   
   * **第一步**：AI生成3-5个探索视角卡片（读一本书、看一个视频、写一篇思考、找人讨论……）。用户可多选，也可跳过直接创建。
   
   * **第二步**：为每个选中视角生成一个具体的第一步行建议（含搜索口令）。用户可调整或跳过。
   
   * **第三步**：汇总展示即将创建的烈焰草稿（一团或多团）。每团烈焰预填充标题、推荐理由、探索口令。用户可修改、删除某团烈焰、或添加新的。
   
   * **草原选择器**：带搜索的输入框。输入时实时匹配已有草原。输入新名称则自动创建新草原。保存时校验重名。

3. 用户确认保存。批量创建烈焰（`status: burning`）。火种保持原样。

4. 弹出平台跳转面板，探索口令已复制到剪贴板。

#### 流程B：完成燃烧与取火

1. 用户在草原或野火中，对一团烈焰点击「完成燃烧」。

2. 弹出记录弹窗，**必填**（可为空格或空字符）。用户写感想，保存。

3. 感想若为空或仅空格，烈焰详情展示为“默默燃尽”。

4. 保存同时触发**取火**：
   
   * 检查取火限制：`rekindleCount >= 10` 则禁止；距上次取火需 >= 5秒。
   
   * 更新烈焰状态（`status=burned`, `userRecord`, `completedAt`）。
   
   * 跳转取火页。

5. 取火页交互：
   
   * 上区：手动创建火种表单。
   
   * 下区：显示“AI正在帮你取火中...”，调用取火接口。
   
   * 返回后展示3粒火种卡片，每粒有「保留」和「丢弃」按钮。
   
   * 底部「重新取火」（同样受5秒冷却和10次上限限制）。
   
   * 用户保留的火种即时入库（`sourceType=ai_rekindle`, `sourceFlameId`, `sourcePrairieId`）。
   
   * 可展开“历史取火记录”。
   
   * 关闭或返回：已保留火种入库，未操作的结果不保存。

6. **取火记录保存**：每次取火创建一条 `rekindleRecord`。

#### 流程C：整理（野火 → 草原）

1. 用户在草原Tab点击「整理」。

2. 收集所有 `prairieId IS NULL` 且 `isDeleted=false` 的烈焰，调用整理接口。

3. 展示**分组总览**列表：
   
   * “X团烈焰 → 归入「草原名」”（action=merge）
   
   * “X团烈焰 → 创建新草原「建议名」”（action=create）

4. 用户点击任一列表项，进入**逐条确认**：
   
   * 展示受影响烈焰标题列表和AI归类理由。
   
   * 按钮：「确认归入」、「跳过」、「手动调整」。

5. 确认后执行数据库更新。

6. 用户可随时退出，已确认生效，未确认下次再分析。

7. **整理入口提示**：用户完成第3次取火后，野火区底部显示轻量提示：“你的探索已经蔓延了一小片，要整理一下吗？”点击进入整理页。

#### 流程D：探索页

1. 用户进入探索页（底部导航中间按钮或草原内入口）。

2. **无输入时**：展示推荐内容——基于已有草原的深入方向、跨领域随机组合。

3. **有输入时**：用户输入关键词（如“咖啡”），AI围绕关键词生成多角度火种（历史背景、科学原理、实践入门、文化现象、相关书影音）。

4. 每粒火种标注视角类型，可「加入火盆」或直接「开始点燃」进入点燃三步向导。

#### 流程E：火种动态与缓存池

1. 系统初始预置5粒火种（`sourceType=ai_seed`）。

2. 后台维护 `seedBuffer` 表，保持 `used=false` 的火种 >= 10粒。

3. 低于10粒时，后台静默调用种子补充接口，批量生成5粒。

4. 用户点「换一批」：从 `seedBuffer` 取5粒 `used=false` 火种，标记 `used=true`，导入 `sparks`。即时响应，无等待。

五、数据模型
------

### 5.1 数据库：`kindling`（IndexedDB）

#### 表：`sparks`

| 字段                | 类型                                                   | 说明                 |
| ----------------- | ---------------------------------------------------- | ------------------ |
| `id`              | string (UUID)                                        | 主键                 |
| `content`         | string                                               | 火种文本内容             |
| `sourceType`      | enum: `user`, `ai_rekindle`, `ai_seed`, `ai_explore` | 来源                 |
| `sourceFlameId`   | string?                                              | 来源烈焰ID（取火时记录）      |
| `sourcePrairieId` | string?                                              | 来源草原ID（取火时记录）      |
| `sourceSparkId`   | string?                                              | 来源火种ID（点燃时记录，用于追溯） |
| `createdAt`       | number                                               | 创建时间               |
| `isDeleted`       | boolean (default false)                              | 逻辑删除               |

**火盆查询：** `sparks.where({ isDeleted: false }).reverse().sortBy('createdAt')`

#### 表：`flames`

| 字段                     | 类型                        | 说明                   |
| ---------------------- | ------------------------- | -------------------- |
| `id`                   | string (UUID)             | 主键                   |
| `title`                | string                    | 烈焰标题                 |
| `description`          | string                    | 烈焰详细描述（可选）           |
| `recommendationReason` | string                    | 推荐理由                 |
| `searchPhrase`         | string                    | 探索口令                 |
| `status`               | enum: `burning`, `burned` | 燃烧中 / 已燃尽            |
| `prairieId`            | string?                   | 所属草原ID（null则为野火）     |
| `sourceSparkId`        | string?                   | 来源火种ID               |
| `igniteBatchId`        | string?                   | 点燃批次ID（同一批点燃的多团烈焰共享） |
| `userRecord`           | string                    | 用户感想（强制，可为空字符）       |
| `completedAt`          | number?                   | 完成时间戳                |
| `createdAt`            | number                    | 创建时间                 |
| `isDeleted`            | boolean (default false)   | 逻辑删除                 |
| `rekindleCount`        | number (default 0)        | 已完成取火次数              |
| `lastRekindleTime`     | number?                   | 上次取火时间戳              |

**野火查询：** `flames.where({ prairieId: null, isDeleted: false }).reverse().sortBy('createdAt')`  
**草原下烈焰：** `flames.where({ prairieId: 'xxx', isDeleted: false }).reverse().sortBy('createdAt')`

| 字段            | 类型                         | 说明         |

| `createdAt`   | number                     | 创建时间       |

**索引：** `name` (唯一)  
**删除规则：** 用户点删除 → `status = archived`，关联烈焰不变。用户可在存档区恢复。彻底删除需二次确认，物理删除记录，关联烈焰 `prairieId = null`。

#### 表：`seedBuffer`

| 字段          | 类型                      | 说明       |
| ----------- | ----------------------- | -------- |
| `id`        | string (UUID)           | 主键       |
| `content`   | string                  | 火种文本内容   |
| `createdAt` | number                  | 生成时间     |
| `used`      | boolean (default false) | 是否已展示给用户 |

#### 表：`deviceInfo`

| 字段             | 类型                | 说明       |
| -------------- | ----------------- | -------- |
| `id`           | string ("device") | 固定主键     |
| `deviceUuid`   | string            | 设备唯一标识   |
| `lastSyncTime` | number?           | 上次云端备份时间 |

### 5.2 数据流转图

text

用户手写 → sparks (isDeleted=false)
seedBuffer + "换一批" → sparks (isDeleted=false)
探索页AI生成 → sparks (isDeleted=false)
sparks + 点燃 → flames (status=burning, prairieId=建议或null, sourceSparkId, igniteBatchId)
flames + 完成燃烧 → flames (status=burned, userRecord, completedAt)
                 → rekindleRecords (sparksResult, retainedSparkIds)
                 → sparks (新的 ai_rekindle 火种, sourceFlameId, sourcePrairieId)
prairies + 整理 → flames.prairieId 更新
prairies 存档 → prairie.status = archived, flames.prairieId 不变
prairies 物理删除 → prairie 删除, flames.prairieId = null (回野火)
sparks / flames 删除 → isDeleted = true
六、AI接口定义
--------

### 6.1 通用原则

* **AI Provider:** MiniMax (MiniMax-Text-01，兼容 OpenAI Chat Completions API)

* **后端代理:** Cloudflare Workers 转发，保护 API Key

* **输出格式:** 严格 JSON，Prompt 末尾约束输出格式

* **降级策略:** 所有接口失败时静默降级，不阻塞用户流程

* **Prompt 约束:** 不使用产品隐喻术语，用最直白语言描述任务

### 6.2 接口一：点燃辅助

**端点:** `POST /api/ai/ignite`  
**描述:** 将用户的灵感想法转化为具体的探索行动建议。

**输入:**

json

{
  "sparkContent": "用户的想法或问题文本",
  "existingPrairies": ["已有草原名1", "已有草原名2"]
}

**MiniMax Prompt:**

text

用户有一个想法或问题，想把它变成具体的探索行动。请按以下三步帮助用户：
第一步：为这个想法生成3-5个不同的探索视角。视角类型可以包括：读一本书或文章、看一个视频或纪录片、写一篇自己的思考、找人讨论或采访、动手实践或体验。
第二步：为每个视角推荐一个具体的、低门槛的第一步行动建议。每个行动附上一个可在B站、知乎、小红书等平台搜索的自然语言短语。
第三步：汇总为一份行动草稿。
用户的想法：{sparkContent}
用户已有的草原：{existingPrairies}
返回格式（严格JSON，不要任何额外文字）：
{
  "perspectives": [
    {
      "type": "视角类型（如：阅读/观看/写作/讨论/实践）",
      "description": "视角描述，1-2句话",
      "firstStep": "具体的第一步行动建议",
      "searchPhrase": "可在内容平台搜索的自然语言短语"
    }
  ],
  "suggestedPrairie": "从已有草原中选最匹配的，或null",
  "newPrairieSuggestion": "如果都不匹配，建议一个新草原名（简要具体，如"咖啡拉花入门"），或null"
}

**输出:** JSON对象，包含 `perspectives` 数组及各视角详情。

**降级:** 若调用失败，前端展示基础表单，用户手动创建烈焰。

### 6.3 接口二：取火

**端点:** `POST /api/ai/rekindle`  
**描述:** 用户完成一个探索任务后，生成3个新问题或灵感。

**输入:**

json

{
  "taskTitle": "任务标题",
  "taskDescription": "任务描述（可为空）",
  "userRecord": "用户感想（可为空字符串）",
  "sourcePrairie": "所属草原名（可为空字符串）"
}

**MiniMax Prompt:**

text

用户完成了一个探索任务，并写下了感想。请生成3个新的问题或灵感，帮助用户继续探索相关方向。
任务标题：{taskTitle}
任务描述：{taskDescription}
用户感想：{userRecord}
所属草原：{sourcePrairie}
每个新问题或灵感应该：

- 与用户的探索内容紧密相关
- 提供新的视角或切入点，而不是总结已完成的任务
- 是用户可以进一步探索的方向（比如：读了书，可以建议看相关纪录片、玩游戏、听播客、实地探访）
- 每个问题或灵感控制在1-2句话，自然、流畅、不刻意
  返回格式（严格JSON，不要任何额外文字）：
  {
  "newQuestions": ["问题1", "问题2", "问题3"]
  }

**输出:** JSON对象，包含3个问题/灵感字符串数组。

**降级:** 若调用失败，取火页仅展示手动创建火种表单，提示“AI暂时无法取火，你可手动创建火种”。

### 6.4 接口三：整理

**端点:** `POST /api/ai/organize`  
**描述:** 分析未分类的探索任务，给出归类建议。

**输入:**

json

{
  "unclassifiedTasks": [
    { "title": "读《局外人》" },
    { "title": "看纪录片《赫尔佐格》" }
  ],
  "existingPrairies": ["存在主义探索", "咖啡入门实践"]
}

**MiniMax Prompt:**

text

用户有一些尚未分类的探索任务，以及一些已有的草原。请分析这些未分类任务，判断它们是否可以归入已有草原，或围绕某个新方向创建新草原。
未分类任务列表：{逐一列出title}
已有草原：{existingPrairies}
规则：

- 如果某些任务与某个已有草原在主题上高度相关，建议归入该草原
- 如果某些任务围绕一个共同的新方向（像"咖啡拉花入门"、"存在主义文学探索"这样具体，不能太宽泛如"哲学"，也不能太窄如"某一个问题"），建议创建一个新草原
- 只对能明确匹配的任务做出建议。不确定的保持沉默
- 同一个任务不能出现在两个建议中
  返回格式（严格JSON，不要任何额外文字）：
  {
  "suggestions": [
    {
  
      "action": "merge",
      "taskIndices": [0, 2],
      "targetPrairie": "已有草原名",
      "reason": "简短理由"
  
    },
    {
  
      "action": "create",
      "taskIndices": [1, 3],
      "newPrairieName": "建议草原名",
      "reason": "简短理由"
  
    }
  ]
  }

**输出:** JSON对象，包含建议数组。

**降级:** 若失败，前端提示“整理暂不可用”，用户可手动归类。

### 6.5 接口四：探索生成

**端点:** `POST /api/ai/explore`  
**描述:** 根据用户输入的关键词或问题，生成多角度的探索火种。

**输入:**

json

{
  "query": "用户输入的关键词或问题",
  "existingPrairies": ["已有草原名1", "已有草原名2"]
}

**MiniMax Prompt:**

text

用户对以下关键词或问题感兴趣，想进行探索。请围绕这个主题，生成5-8粒火种（灵感/问题/探索方向）。
用户输入：{query}
用户已有草原：{existingPrairies}
每粒火种应该：

- 指向一个具体、可探索的知识缺口或行动灵感
- 从不同角度切入（历史、科学、文化、实践、批判视角等）
- 标注这个火种的视角类型（阅读/观看/实践/讨论/思考）
- 每个火种控制在1-2句话，能引发好奇心
  返回格式（严格JSON，不要任何额外文字）：
  {
  "sparks": [
    {
  
      "content": "火种文本",
      "type": "视角类型"
  
    }
  ],
  "suggestedPrairie": "从已有草原中选最匹配的，或null",
  "newPrairieSuggestion": "如果都不匹配，建议一个新草原名，或null"
  }

**输出:** JSON对象，包含火种数组。

**降级:** 若失败，提示用户稍后重试。

### 6.6 接口五：种子缓存补充

**端点:** `POST /api/ai/seed-buffer`  
**描述:** 批量生成火种，补充缓存池。

用户近期关注：{recentSparks}
用户已有草原：{existingPrairies}
每个问题应该：

- 指向一个具体、可探索的知识缺口（像"唐僧取的真经到底讲了什么？"这样）
- 能引发好奇心，让人想去搜索了解

  返回格式（严格JSON，不要任何额外文字）：

七、探索口令与平台跳转
-----------

### 7.1 探索口令定义

“探索口令”是点燃辅助接口中生成的 `searchPhrase` 字段——一句自然语言短语，用户可直接粘贴到内容平台搜索框。

### 7.2 使用流程

2. 弹出平台跳转面板（B站、知乎、小红书等图标）。

3. 用户点击图标 → 拼接搜索URL → 新标签页打开。

### 7.3 平台URL模板

八、业务规则约束汇总
----------

| 场景          | 规则                                                 |
| ----------- | -------------------------------------------------- |
| **点燃产出**    | 一个火种可点燃产生1~N团烈焰                                    |
| **取火冷却**    | 同团烈焰两次取火间隔 ≥ 5秒（前端置灰+后端校验）                         |
| **取火上限**    | 同团烈焰最多取火10次                                        |
| **草原名唯一**   | 前端+后端双重校验                                          |
| **草原删除**    | 操作为存档（`status=archived`），烈焰关联不变；彻底删除需二次确认          |
| **火种/烈焰删除** | 逻辑删除（`isDeleted=true`），火种不影响已点燃烈焰，烈焰不影响已产生火种       |
| **点燃关联**    | 火焰记录来源火种和点燃批次（`sourceSparkId`, `igniteBatchId`）    |
| **取火关联**    | 新火种记录来源烈焰和来源草原（`sourceFlameId`, `sourcePrairieId`） |
| **感想空值**    | 前端展示为“默默燃尽”                                        |
| **换一批**     | 从 `seedBuffer` 即时取5粒，缓存池 < 10时后台补充                 |
| **备份**      | 纯手动，全量JSON上传至阿里云OSS                                |

十、MVP边界与未来规划
------------

### 10.1 MVP包含

* 火盆（火种CRUD、种子缓存池、换一批）

* 探索页（关键词探索、火种生成）

* 取火（AI生成3粒+手动创建+保留/丢弃+冷却/上限）

### 10.2 MVP不做什么

* 用户注册/登录、云端同步

* 社交/分享/发现

* 无力感模式「余烬」

* 浏览器扩展/系统分享菜单

* AI自动归类（整理需手动触发）

* 草原叙事摘要（AI总结）

* 草原合并

* 取火时跨草原链接

### 10.3 未来规划

* **二期**：用户系统、云端同步、草原分享/发现、余烬模式、草原叙事摘要、浏览器扩展

* **远期**：社区生态、付费探索路径、跨用户知识图谱、「生活方式开源平台」

* **终极愿景**：让充满生命力的生活方式，成为一种可实践、可传播的社会实验

十一、思考与决策记录
----------

本章记录设计过程中做出的关键取舍及其原因，供后续迭代参考。

| 决策                      | 原因                                    | 可能的迭代方向               |
| ----------------------- | ------------------------------------- | --------------------- |
| 火种与草原都保留，不做合并           | 火种是瞬时聚合（一次点燃多团烈焰），草原是长期叙事（跨火种、跨时间的脉络） | 未来可增加草原叙事摘要强化草原独特性    |
| 草原删除改为存档而非物理删除          | 草原承载叙事，删除则链接断裂，与对抗虚无的核心定位矛盾           | 存档后的草原可在后续版本支持合并      |
| MVP不做草原叙事摘要             | 时间轴已展示脉络，AI总结是锦上添花；需控制MVP范围           | 二期引入                  |
| 取火不引入跨草原链接              | 取火聚焦当前烈焰的延续，跨草原链接与整理功能重叠              | 整理功能后续可增加跨草原链接发现      |
| 点燃改为三步向导（选视角→选行动→确认）    | 降低从灵感到行动的认知门槛，拆解“点燃”黑箱                | 观察用户是否大量跳过前两步，决定是否简化  |
| 探索页共存两种模式（有输入/无输入）      | 以用户意图为起点，而非让用户选择模式，降低认知负担             | 观察哪种模式的火种转化率更高        |
| 「换一批」从缓存池取而非实时AI生成      | 零等待，体验流畅                              | 缓存池策略可优化：基于用户行为调整生成偏好 |
| 设备UUID + OSS手动备份，不做用户系统 | MVP单人使用，极简方案零成本                       | 二期引入用户系统和自动同步         |
| 记录强制填写但可为空格             | 守护闭环仪式感，但尊重用户“无话可说”的状态                | 已优化为“默默燃尽”展示          |
